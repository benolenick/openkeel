# Critique — Calcifer's Ladder Design

**Date:** 2026-04-08
**Target:** [`../calcifer_ladder_design_2026-04-08.md`](../calcifer_ladder_design_2026-04-08.md)
**Reference implementation studied:** Claude Code source snapshot in `/home/om/Downloads/claude-code-main.zip`

---

## Frame

The current Ladder design has the right top-level ambition:

**Use a supervisory loop to decide how much intelligence to spend on a step, and only invoke a full Claude-style agent loop when the step actually needs it.**

The main weakness is structural:

**the current design over-focuses on runner routing and under-specifies loop state, evaluation, and context lifetime.**

Claude Code does not work because it has a strong model picker. It works because it has:

- a durable session owner
- a stateful query/tool loop
- typed tools
- permission handling
- persistence and recovery
- explicit mode boundaries

Those responsibilities must exist in Calcifer too, even if the implementation is much thinner.

## What Claude Code Gets Right

From the snapshot, the useful architecture patterns are:

### 1. One session owner

`QueryEngine` owns conversation state, usage, permission denials, file cache, and session persistence for the entire turn.

Relevant files:
- `/tmp/claude-code-main/claude-code-main/src/QueryEngine.ts`
- `/tmp/claude-code-main/claude-code-main/src/query.ts`

This is the right model for Calcifer: the loop needs a durable state owner.

### 2. One generic query/tool loop

The core loop is:

1. build prompt/context
2. ask model
3. inspect tool calls
4. execute typed tools
5. append normalized results
6. continue until terminal condition

That is the real engine. The specific model is only one variable inside it.

### 3. Typed tool surface

Claude Code does not let the model improvise execution. It gives the model typed tools with contracts and a shared execution context.

Relevant files:
- `/tmp/claude-code-main/claude-code-main/src/Tool.ts`
- `/tmp/claude-code-main/claude-code-main/src/tools.ts`

### 4. Headless structured runtime

The CLI is not only a TUI. It has a structured stdin/stdout runtime for headless operation and control flow.

Relevant file:
- `/tmp/claude-code-main/claude-code-main/src/cli/structuredIO.ts`

This matters because Calcifer should be a runtime, not just a terminal UX.

### 5. Mode distinctions still exist

Claude Code has one overall engine, but still distinguishes:

- fast-path startup
- interactive REPL mode
- headless print mode
- bridge / remote modes
- background sessions

That is the right lesson for Ladder:

**one engine does not mean one undifferentiated mode.**

## Main Critique Of The Current Ladder Design

### 1. The router is too central

The current design treats `router.pick(job)` as the centerpiece.

That is the wrong emphasis.

Routing matters, but the loop shell matters more. In practice, failures will come from:

- weak context ownership
- weak evaluator semantics
- ambiguous task transitions
- bad permission handling
- poor context shrinking

before they come from cost ranking mistakes.

### 2. `Job` is too coarse

The current `Job` abstraction tries to cover:

- planning
- tool-output shrinking
- local replacement
- answering
- judging

That is too much for one object.

The system needs at least:

- a step spec
- an evaluator spec
- a turn/task state object

### 3. `success_check` is too weak as free text

The design makes `success_check` carry too much of the trust model.

It should become a structured evaluator contract with fields like:

- expected artifact type
- deterministic verifier to run
- fallback judge type
- allowed judge runners
- escalation policy on inconclusive verdict

### 4. The state owner is missing

The current loop pseudocode only visibly carries:

- `messages`
- `job`
- `iterations`

That is too thin.

Calcifer needs a real `TurnState` that owns:

- session/task identity
- live messages
- tool result lineage
- current plan / directive
- permission context
- budget state
- retained artifacts
- compaction / shrinking state
- runner history

### 5. The ladder should remain explicit

The current design throws out the rung model and says the ladder is implicit in cost ordering.

That is too aggressive.

You still have distinct execution modes:

- passthrough
- shrink-only
- replace-safe
- planner-required
- judge-required

Those are rungs in all but name. Keep them explicit as execution modes or escalation states.

### 6. Benchmarks are too one-shot

The proposed probe suite tests summaries, QA, code edits, and judges.

Useful, but not enough.

You also need to benchmark bounded tool-loop reliability:

- can this runner complete a 2-4 step CLI tool loop?
- does it ask for the right tool next?
- does it preserve step intent after tool output?
- does it terminate cleanly?

That is much closer to the real failure mode of local runners.

### 7. `maybe_shrink` is a hidden subsystem

Shrinking should not be a casual helper inside the loop. It should be an explicit context-budget subsystem with policy and provenance.

Otherwise the system will silently trade away verifiability for token savings.

## What Should Change

The design should be reframed around:

1. **Outer supervisory loop**
   - owns tasks, escalation, budgeting, context lifetime, and final judgment

2. **Inner execution loops**
   - deterministic path
   - lightweight local reasoning path
   - full Claude-style agent loop

3. **Explicit state primitives**
   - turn
   - task
   - task session
   - step spec
   - evaluator spec

4. **Explicit context lifetimes**
   - ephemeral turn context
   - bounded task context
   - long-term retrieved memory

5. **Explicit execution modes**
   - direct
   - semantic
   - local bounded loop
   - strong full agent
   - planner / judge / delegate

---

## Bottom Line

The current design is pointed in the right direction but is still too much of a router design and not enough of an agent-runtime design.

The right redesign is:

**Calcifer is a supervisory CLI runtime that chooses between multiple execution modes, one of which is a Claude-style inner agent loop.**

That keeps the adaptive-cost vision while preserving the real lessons from Claude Code.
